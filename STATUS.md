## План — перенос во внутренний контур банка ⚠️ НЕ НАЧИНАТЬ БЕЗ ЯВНОГО ЗАПРОСА

Подготовка к будущему переносу проекта во внутренний контур банка. Это бэклог на будущее, а не текущая задача — **не приступать к реализации ни одного пункта**, пока пользователь явно не попросит перейти к конкретному пункту в новой сессии.

- **Docker для продакшна** — `Dockerfile` с multi-stage сборкой (build-стадия: `npm ci` + `npm run build`; финальная стадия — только `dist/` + `scripts/dev-api-server.ts`/прод-сервер под node, минимальный рантайм-образ).
- **README — переменные окружения** — задокументировать в `README.md` все переменные из `.env.local` (назначение, обязательность, пример значения): ключи Anthropic/прокси, `PROXY_SECRET`, `ANTHROPIC_PROXY_URL`, `BRAVE_PROXY_URL`, порты, авторизационные секреты и т.д.
- **Доступ к моделям через OpenAI API-совместимый формат** — переписать слой обращения к ИИ (сейчас `@anthropic-ai/sdk` в `scripts/dev-api-server.ts`) на OpenAI-совместимый API, чтобы в контуре банка можно было подключить локальную модель Qwen (без завязки на Anthropic/Cloudflare Worker прокси).
- **Системные промпты — вынести в папку скиллов** — промпты ассистентов (сейчас `src/lib/agentPrompts.ts`) разнести по отдельным файлам в папке скиллов (по агенту/сценарию), а не хранить одним модулем.
- **Логирование и аудит-события (требование ИБ)** — по требованиям ИБ банка любое действие пользователя, изменяющее конфигурацию (настройки профиля, интеграций, агентов/сервисов, прав доступа и т.п.), должно логироваться как событие аудита (кто, когда, что изменил, старое/новое значение). Нужен единый audit-log слой на API-сервере (отдельно от обычных логов приложения) с последующей интеграцией в централизованное логирование банка (ELK/Splunk).

РЕШЕНО (реализация на будущее): единый self-routing ассистент, БЕЗ оркестратора и БЕЗ отдельного классификатора.** Выбрано под 3 приоритета переноса на локальную Qwen: (1) phase-1 тест «повторит ли Qwen то, что делает Haiku» — берём максимально близкую к текущей архитектуру, чтобы тест был честным; (2) нагрузка (десятки продактов с утра одновременно) важнее токенов — минимум вызовов модели на реплику (оркестратор/классификатор = кратно больше forward-pass'ов = хуже пропускная способность); (3) надёжность для любых продактов.   - **Тяжёлые спеки форматов (QBR HTML+SVG+CSS `agentPrompts.ts:139-155`, детали CJM) — из system-промпта в результат инструмента** (напр. `start_qbr_report` возвращает шаблон). Держит общий prefix маленьким и стабильным; спек приходит только когда путь выбран.
  - **Опора на prefix caching vLLM/SGLang** (локальный аналог A1): один стабильный промпт+набор инструментов на всех → общий префикс считается один раз для всех юзеров. ⚠️ Это ОТМЕНЯЕТ идею динамического скоупа инструментов из A2 — разные наборы под агента = cache miss = хуже под нагрузку. Держим набор инструментов стабильным. ⚠️ Волатильный контент (дата/имя юзера) — НЕ в начало промпта, иначе префикс «плывёт» и кэш промахивается на каждый запрос.
  - Надёжность интента гарантируется тест-сетом (Vitest): размеченные фразы юзера → ожидаемый инструмент/поведение.
  - Подробная история обсуждения и отвергнутые варианты — в `agent-architecture-notes.txt` (не читается при старте сессии).
