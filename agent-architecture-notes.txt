0. КОНТЕКСТ / ЗАЧЕМ ВООБЩЕ ОБСУЖДАЛИ
--------------------------------------------------------------------------------
Проект скоро переносится во внутренний контур банка с подключением ЛОКАЛЬНОЙ
модели Qwen (~30-32B, точного «35B» нет: есть Qwen2.5-32B, Qwen3-32B,
Qwen3-30B-A3B MoE) вместо текущего Anthropic Haiku.

Пошли от конкретной находки инвентаризации (пункт A2):
инструменты (все 15, TOOL_DEFINITIONS) отправляются модели НА КАЖДЫЙ запрос,
независимо от активного агента. Отсюда родился вопрос про скоупинг инструментов,
а из него — большой разговор про то, чем вообще должны быть «агенты».

ВАЖНОЕ ФАКТИЧЕСКОЕ УТОЧНЕНИЕ (всплыло в обсуждении):
- system-промпт УЖЕ скоупится — уходит base + промпт ОДНОГО активного агента
  (AIPanelSider.tsx:1328-1330), а НЕ промпты всех агентов. Это частое
  заблуждение. Целиком/всегда уходят только ИНСТРУМЕНТЫ.
- Текущий цикл (AIPanelSider.tsx:1320) УЖЕ агентный: модель вызывает инструменты
  и итерирует до stop_reason != tool_use. То есть агенты — это не «просто
  промпты», это промпт + общий агентный цикл + общий (полный) набор инструментов.

--------------------------------------------------------------------------------
1. КАКИЕ ВАРИАНТЫ РАССМАТРИВАЛИ
--------------------------------------------------------------------------------
Спектр «насколько агент — полноценный агент», от малого к большому:

Вариант 1 — Пресеты промптов (КАК СЕЙЧАС).
  Агент = system-промпт + общие инструменты + общий цикл. «Роутинг» = юзер
  вручную выбирает промпт.

Вариант 2 — Скоупленные агенты (капсулы).
  Каждый агент = промпт + СВОЙ набор инструментов + свой контракт вывода.
  Одна модель, один цикл. Небольшое изменение поверх текущего.

Вариант 3 — Оркестратор + специалисты (мультиагент), паттерн «агент-как-инструмент».
  Супервайзер, чьи ИНСТРУМЕНТЫ — это сами специалисты (ask_metrics_agent, ...).
  Модель-оркестратор сама решает кого вызвать, может вызвать нескольких по
  цепочке (метрики -> задача) и синтезировать. ВАЖНО: этот вариант ПОГЛОЩАЕТ
  роутинг — отдельный классификатор не нужен, «маршрутизация» = обычный выбор
  инструмента. Изначально я склонялся сюда.

Вариант 4 — Автономные агенты с памятью и хендоффом (OpenAI Swarm-стиль).
  Долгоживущие сущности, передают управление друг другу, фон. Максимум мощности
  и сложности. Для нашей задачи избыточно.

Отдельная под-идея (обсуждали до вариантов) — АВТО-РОУТИНГ через отдельный
лёгкий КЛАССИФИКАТОР: дешёвый вызов модели «верни одну метку агента», потом
основной вызов с этим агентом. Плюс sticky-состояние для многоходовых агентов,
плюс ручной override. Юзер хотел: оставить ручной выбор, но под капотом
авто-роутинг, «главное чтоб работал надёжно».

--------------------------------------------------------------------------------
2. КЛЮЧЕВЫЕ ПРОБЛЕМЫ/ЛОВУШКИ, КОТОРЫЕ ВЫЯВИЛИ
--------------------------------------------------------------------------------
(A) МНОГОХОДОВЫЕ АГЕНТЫ С СОСТОЯНИЕМ (человек-в-цикле). Самая коварная.
    QBR: показывает metric-selector -> ЖДЁТ клик юзера -> генерит HTML-отчёт.
    CJM Режим 0: предлагает варианты -> ЖДЁТ выбор -> Режим 1 создаёт карту.
    - Наивный классификатор, роутящий КАЖДОЕ сообщение, между ходами переключит
      агента (юзер прислал «выбрал метрики id1,id2» — классификатор решит, что
      это агент метрик) и СЛОМАЕТ сценарий. => нужен sticky.
    - Паттерн «агент-как-инструмент» (Вар. 3) плохо ложится на такие агенты:
      sub-agent обычно «запрос->ответ», а тут надо ПОСРЕДИ вернуть UI юзеру.
    - ГЛАВНОЕ: это проблема АРХИТЕКТУРНАЯ, НЕ зависит от размера модели.

(B) ЦЕЛЕВАЯ МОДЕЛЬ — Qwen ~32B, не Claude.
    - Единый мега-промпт (модель сама роутит выбором инструментов): элегантно на
      Claude, но на слабее модели 9 наборов инструкций в одном промпте могут
      дать деградацию форматов вывода.
    - Явный классификатор (узкая задача «1 из N») — проще для слабой модели.
    - НО: Qwen 32B — НЕ слабая модель, function-calling у неё нормальный.
      Я изначально перестраховался с «модель не потянет оркестратор».

(C) «НАДЁЖНО» НЕЛЬЗЯ ГАРАНТИРОВАТЬ БЕЗ ТЕСТА.
    Единственный честный способ — размеченный тест-сет (30-50 фраз юзера ->
    ожидаемый агент/инструмент) как Vitest-тест. Тогда «работает надёжно» = зелёный
    npm test, и регрессии при правке промпта видны сразу. Vitest в проекте уже есть.

3. ЧТО ПЕРЕВЕРНУЛО РЕШЕНИЕ — 3 ПРИОРИТЕТА ЮЗЕРА
--------------------------------------------------------------------------------
Юзер дал вводные, которые изменили ответ (развернули ОТ оркестратора):

Приоритет 1 — PHASE-1 ТЕСТ СПОСОБНОСТЕЙ.
  Первый этап в контуре — проверить, сможет ли локальная Qwen повторить то, что
  сейчас делает Haiku, и насколько успешно.
  => нужна архитектура, МАКСИМАЛЬНО БЛИЗКАЯ к текущей. Оркестратор смазал бы
     сигнал теста (если сломается — непонятно, модель виновата или слой
     оркестрации).

Приоритет 2 — НАГРУЗКА (связывающее ограничение!).
  Если дойдёт до релиза — десятки продактов заходят С УТРА одновременно и
  начинают работать с ассистентами. Токены НЕ важны (своё железо), важна
  НАГРУЗКА на инференс, чтоб модель выдержала.
  => нагрузка = ЧИСЛО ПРОХОДОВ МОДЕЛИ (каждый вызов = forward pass = занятый
     GPU-слот) + объём prefill. Оркестратор/классификатор = +1..+N вызовов на
     запрос = при 50 юзерах кратно меньше пропускная способность.
     ЭТОТ ПРИОРИТЕТ НАПРЯМУЮ УБИВАЕТ ОРКЕСТРАТОР. Самое дешёвое по нагрузке —
     ОДИН вызов на реплику, где одна модель сама понимает и сама выбирает
     инструменты (self-routing).

Приоритет 3 — НАДЁЖНОСТЬ ДЛЯ ЛЮБЫХ ПРОДАКТОВ.
  Должен одинаково хорошо работать и для продвинутых, и для тех, кто в ИИ не
  разбирается. Если «тупой» продакт напишет «хочу cjm» или «бриф на исследование»
  и не получит ожидаемого — решит, что сервис тупой, и уйдёт.
  Конкретный кейс: продакт выбрал агента Трендвотчер, забыл про это, потом просит
  поставить задачу — должно всё равно сработать.

--------------------------------------------------------------------------------
4. РАЗВОРОТ ПРО PREFIX CACHING (важно — отменяет прежний совет по A2)
--------------------------------------------------------------------------------
Раньше советовал ДИНАМИЧЕСКИ резать инструменты под агента (экономия ТОКЕНОВ).
Под НАГРУЗКУ на локальной Qwen калькуляция ПЕРЕВОРАЧИВАЕТСЯ:

- vLLM/SGLang (стандартный стек под Qwen) умеют AUTOMATIC PREFIX CACHING:
  одинаковый префикс (system + инструменты) считается ОДИН раз и переиспользуется
  всеми запросами и всеми юзерами; KV-кэш общего префикса хранится один на всех,
  не по копии на юзера.
- => СТАБИЛЬНЫЙ, одинаковый для всех промпт+инструменты = почти нулевой prefill
  на общую часть у всех утренних юзеров.
- => ДИНАМИЧЕСКИЙ скоуп под агента = у каждого свой префикс = cache miss =
  prefill заново = ХУЖЕ под нагрузкой.
ВЫВОД: держим ОДИН стабильный набор инструментов и один стабильный промпт;
дешевизну даёт prefix-кэш, а не скоупинг. (Локальный аналог пункта A1.)

⚠️ Волатильный контент (дата/имя юзера) НЕ в начало промпта — иначе префикс
«плывёт» и кэш промахивается на каждый запрос. Сейчас base-промпт кладёт дату
наверх (agentPrompts.ts:5-11) — бьёт кэш раз в сутки (терпимо), но время/юзера
туда добавлять НЕЛЬЗЯ. При переписывании — волатильное в конец / в user-сообщение.

--------------------------------------------------------------------------------
5. ИТОГОВОЕ РЕШЕНИЕ
--------------------------------------------------------------------------------
ЕДИНЫЙ SELF-ROUTING АССИСТЕНТ. БЕЗ оркестратора. БЕЗ отдельного классификатора.
Практически текущая архитектура, но с 3 правками:

(1) Ручной выбор агента -> МЯГКАЯ ПОДСКАЗКА, а не жёсткий режим.
    Выбор лишь добавляет акцент в контекст, НЕ отключает остальные инструменты.
    Решает кейс «выбрал Трендвотчера, забыл, просит задачу»: инструмент создания
    задачи никуда не делся -> задача ставится. Залипать не в чем, жёстких режимов
    больше нет. (Закрывает приоритет 3 с нулём лишних вызовов.)

(2) Тяжёлые спеки форматов — из system-промпта В РЕЗУЛЬТАТ ИНСТРУМЕНТА.
    QBR HTML+SVG+CSS (agentPrompts.ts:139-155), детали CJM. Пусть тяжёлая
    инструкция приходит как tool-result ровно когда путь выбран (напр.
    start_qbr_report возвращает шаблон). Держит общий префикс маленьким и
    стабильным + меньше нагрузка на генерацию. Это ровно исходная идея юзера про
    CJM: «вписывать в шаблон, а не заставлять модель писать с нуля».

(3) Обрезка/суммаризация истории (пункт A4) — под нагрузкой критичнее.
    Длинный диалог = большой prefill на каждый ход. Старые tool-результаты
    усекать. Влияет на то, сколько юзеров влезет одновременно.

Плюс: надёжность интента — тест-сет на Vitest (размеченные фразы -> ожидаемое
поведение). Нужен при ЛЮБОЙ архитектуре.

КАК ЛОЖИТСЯ НА ПРИОРИТЕТЫ:
  1 (phase-1): самая близкая к текущей архитектура -> честный тест Qwen vs Haiku.
  2 (нагрузка): 1 вызов/реплику + стабильный prefix-кэш = минимум forward-pass'ов.
  3 (надёжность): нет жёстких режимов -> нечему «залипнуть»; интент на одной
     модели, меряется тестами.

--------------------------------------------------------------------------------
6. ЧТО ЯВНО ОТВЕРГЛИ И ПОЧЕМУ (чтобы не возвращаться по кругу)
--------------------------------------------------------------------------------
- Оркестратор / мультиагент (Вар. 3): красивый, поглощает роутинг, но +N вызовов
  на запрос => проигрывает по приоритету 2 (нагрузка). Ещё и смазал бы phase-1 тест.
- Отдельный классификатор перед каждым сообщением: +1 вызов на реплику => та же
  проблема нагрузки; плюс ломает многоходовые агенты без sticky-логики.
- Динамический скоуп инструментов под агента (прежний A2): бьёт prefix-кэш =>
  хуже под нагрузкой. Заменено на «стабильный набор + prefix caching».
- Жёсткие режимы агентов (как сейчас, где выбор запирает контекст): создают кейс
  «залип в неправильном агенте» => противоречит приоритету 3.
