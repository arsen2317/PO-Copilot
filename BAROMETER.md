# Барометр — спецификация проекта

> Единая интеллектуальная среда для продуктовых команд: ИИ-ассистент, автономные агенты и каталог внешних ИИ-сервисов поверх Jira, Confluence, Git, Figma, продуктовых метрик и UX-базы.

Этот файл — источник истины для проекта. Он описывает, **что** мы строим, **как** это устроено и **в каком порядке** реализуем. Держи его в корне репозитория и обновляй при принятии архитектурных решений.

---

## 1. О проекте

**Проблема.** Данные продуктовой команды разорваны между Jira, Confluence, Git, Figma, системами метрик и поддержки. Чтобы понять, что происходит с продуктом, человек вручную собирает картину из 6+ систем. Реакция на проблемы запаздывает, регуляторные риски (compliance) всплывают поздно, time-to-market растёт.

**Решение.** «Барометр» сводит все источники в одну среду и накрывает их тремя ИИ-слоями:

- **Ассистент** — диалоговый интерфейс. Аналитика, генерация артефактов и управление задачами на естественном языке.
- **Агенты** — автономные фоновые модули, которые 24/7 мониторят системы и сами инициируют действия при наступлении условий.
- **Каталог ИИ-сервисов** — управляемый агрегатор сторонних ИИ-инструментов, доступных команде по корпоративной подписке.

Поверх — собственное управление задачами (надстройка над Jira), комнаты для церемоний и база знаний в виде графа продукта.

**Ценность.** Прозрачность процессов, проактивный контроль рисков, сокращение time-to-market, снижение compliance-рисков.

**Контекст.** Продукт проектируется для крупной финансовой организации (банк), отсюда сильный акцент на compliance и аудит.

## 3. Статус и режим разработки

На старте это **frontend-прототип с моковыми данными**. Бэкенд и реальные интеграции — отдельная фаза. Все «ИИ-ответы», срабатывания агентов и метрики на первом этапе генерируются из фикстур, чтобы можно было пройти все ключевые сценарии вживую без внешних систем.

Это сознательное решение: сначала валидируем UX и информационную архитектуру, потом подключаем реальные данные.

## 4. Технологический стек

Дизайн-система проекта — **Ant Design (antd) v5**, интерфейс всегда в **тёмной теме**. Это зафиксированное решение, не рекомендация. Остальное — рекомендация под прототип сложного дашборда; меняй под свои предпочтения, фиксируя решение здесь и в `STATUS.md`.

- **Сборка / язык:** Vite + React + TypeScript
- **Дизайн-система:** Ant Design (antd) v5 — весь интерфейс строим на компонентах antd, не вводим свои аналоги тому, что в antd уже есть
- **Тема:** тёмная по умолчанию через `ConfigProvider` + `theme.darkAlgorithm`; кастомизация через design tokens antd (см. раздел 8)
- **Иконки:** `@ant-design/icons`
- **Графики:** `@ant-design/plots` (воронки, спарклайны, burndown) — остаёмся в экосистеме antd
- **Граф знаний:** `@ant-design/graphs` (на базе G6) — узлы-артефакты и связи
- **Drag-and-drop:** dnd-kit (Канбан, состав спринта, ретро-доска) — в antd нет встроенного DnD
- **Роутинг:** React Router
- **Состояние сервера / данные:** TanStack Query (на этапе моков — обёртка над фикстурами с искусственной задержкой)
- **Локальное состояние:** Zustand для глобального UI-состояния (текущий продукт, открытые панели)
- **Тесты:** Vitest + React Testing Library; Playwright для ключевых сквозных сценариев
- **Хостинг / деплой:** Vercel — preview-деплой на каждый PR, production на `main`; детали в разделе 15

**Принцип:** доступность (a11y) — не опция. Продукт корпоративный; antd закрывает большую часть из коробки, но навигацию с клавиатуры и контраст проверяем.

## 5. Структура проекта

```
src/
  app/                 # роутинг, провайдеры, layout-обёртки
  components/
    layout/            # Header, Sidebar, AppShell
    ui/                # переиспользуемые примитивы (кнопки, карточки, бейджи)
    widgets/           # виджеты дашборда
  features/
    dashboard/
    assistant/
    agents/
    ai-services/
    tasks/
    rooms/
    knowledge-base/
    notifications/
    profile/
  data/
    fixtures/          # моковые данные
    api/               # слой доступа к данным (сейчас читает фикстуры)
    types/             # доменные типы (см. модель данных)
  hooks/
  lib/                 # утилиты (форматирование, скоринг приоритета и т.п.)
  styles/              # глобальные стили, дизайн-токены
```

**Правило:** каждый раздел из навигации = папка в `features/`. Фича владеет своими страницами, компонентами и хуками; общее уезжает в `components/ui` или `lib`.

## 7. Глобальный layout (AppShell)

Каждый экран рендерится внутри общей оболочки.

**Хедер:**
- Логотип слева.
- Глобальный поиск по центру — ищет по задачам, документам, коду, метрикам, диалогам.
- Колокольчик уведомлений с бейджем непрочитанных → открывает выпадающую панель (~400px).
- Аватар с выпадающим меню: профиль, настройки, выход.

**Левая боковая панель:**
- Иконки разделов: Дашборд, Ассистент, Агенты, ИИ-сервисы, Задачи, Комнаты, База знаний.
- Сворачивается до иконок / разворачивается с подписями. Активный раздел подсвечен.
- Внизу — кнопка «Создать» (быстрое создание задачи, агента, дашборда, комнаты).

**Основная область** — содержимое выбранного раздела.

## 8. Дизайн-система (Ant Design, тёмная тема)

Вся визуальная система строится на **antd v5 в тёмной теме**. Цвета, отступы и типографику берём из **design tokens** antd, а не хардкодим литералами.

**Подключение темы** — один `ConfigProvider` в корне приложения:

```tsx
import { ConfigProvider, theme } from 'antd';

<ConfigProvider
  theme={{
    algorithm: theme.darkAlgorithm,
    token: {
      colorPrimary: '#1668dc',   // акцент: ассистент, активные элементы
      colorError:   '#dc4446',   // критические инциденты, просрочка
      colorWarning: '#d89614',   // под риском
      colorSuccess: '#49aa19',   // положительная динамика, готовность
    },
  }}
>
  <App />
</ConfigProvider>
```

Значения токенов выше — стартовые (близки к dark-пресету antd); финальную палитру подбираем под бренд.

**Семантика цветов рисков** (через токены, не литералами):

| Токен antd | Назначение |
|------------|------------|
| `colorPrimary` | ответы ассистента, активные элементы, акценты |
| `colorError` | критические инциденты, просроченные задачи |
| `colorWarning` | задачи под риском |
| `colorSuccess` | положительная динамика метрик, готовность |

**Правила:**
- Тёмная тема — единственная. Светлую не делаем; переключатель темы в профиле пока заглушка.
- Кастомные стили — через `token` и `useToken()`, не через произвольный CSS поверх antd.
- Раскладка — на `Layout`, `Grid`, `Flex` из antd; собственные сетки только там, где antd не покрывает (граф, Канбан).
- Сетка дашборда — настраиваемая, виджеты перетаскиваются (dnd-kit поверх antd `Card`).
- Цвета рисков последовательны во всех разделах (карточки задач, инциденты, бэклог).

## 9. Экраны

### 9.1 Дашборд (`/`)
Первый экран после входа. Сверху — переключатель текущего продукта/команды. Настраиваемая сетка виджетов:

- **Воронка конверсии** — интерактивная диаграмма; при наведении цифры, при клике кнопка «Спросить ассистента».
- **Динамика NPS и обращений в поддержку** — спарклайн за 30 дней.
- **Скорость команды** — burndown текущего спринта + прогноз завершения эпиков.
- **Активные инциденты** — карточки с цветовой индикацией, ведут на страницу инцидента.
- **Последние уведомления** — сводка из центра уведомлений.

Действия: «Добавить виджет» (каталог), перетаскивание, пороговые значения и фильтры. Контекстное меню виджета: открыть в полном размере, настроить, спросить ассистента, удалить.

### 9.2 Ассистент (`/assistant`)
Три колонки:
- **Слева** — список диалогов (Личные / Групповые / Привязанные к задачам). Элемент: иконка, название, последнее сообщение, время. Кнопка «Новый диалог» → выбор контекста (продукт, спринт, задача) и участников.
- **Центр** — лента сообщений. Ответы ассистента на синем фоне, содержат графики, ссылки, карточки-превью. Поле ввода: текст, голос, команды через `/`, упоминания через `@` (агенты, сервисы, коллеги).
- **Справа** — контекст диалога (продукт, спринт, задача), связанные дашборды и изменения в коде, список участников, «Прикрепить артефакт».

### 9.3 Агенты (`/agents`)
Подвкладки: **Готовые агенты**, **Мои агенты**, **Конструктор**.

- **Готовые** — сетка карточек с фильтрами по категориям (название, описание, иконка, популярность). Клик → детальная страница (логика, источники данных, примеры действий, «Добавить в мои»).
- **Мои** — список с переключателем активности. По каждому: статус, последнее срабатывание, число событий за 24ч. Действия: приостановить, настроить, лог, удалить.
- **Конструктор** — визуальный редактор: триггер (расписание, событие Jira, порог метрики, новый PR, новый тикет) → условие (например, «падение конверсии >5% за 1 час») → действие (создать задачу, отправить уведомление, запустить сервис, написать в чат). Внизу — общий лог активности.

### 9.4 ИИ-сервисы (`/services`)
Каталог **сторонних** инструментов, не функций платформы.
- Верхняя панель: поиск + фильтры по категориям (Текст, Дизайн, Код, Аналитика, Исследования, Прототипирование, Автоматизация…).
- Сетка карточек: логотип, название, описание, оценка полезности от коллег, кнопка «Перейти» (новая вкладка).
- Детальный просмотр: описание, примеры, стоимость, наличие корпоративной подписки, комментарии команды.
- Боковая панель: «Избранные» и «Популярное в банке». Кнопка «Предложить сервис».

### 9.5 Задачи (`/tasks`)
Надстройка над Jira. Верхняя панель: переключение представлений, фильтры (статус, исполнитель, спринт, продукт, метки риска), «Создать задачу», быстрый поиск.

- **Канбан** — колонки = статусы Jira; карточки с цветовой индикацией рисков; drag-and-drop; превью при наведении; «Спросить ассистента» на каждой карточке.
- **Список** — сортируемая таблица: ID, заголовок, статус, исполнитель, приоритет, связанные метрики, прогноз влияния, дата завершения.
- **Приоритизированный бэклог** — ранжирование по авто-скору (ICE/RICE + compliance-риски + стратегия); перетаскивание с предупреждениями ассистента о последствиях; панель информации о задаче.

**Детальная страница задачи** — вкладки: Описание, Критерии приёмки, Обсуждение (чат с ассистентом), Compliance, Метрики, История изменений. Автоматически подтянутые артефакты (PR, макеты, Confluence).

### 9.6 Комнаты (`/rooms`)
Список комнат (карточки: название, статус, дата, участники), «Создать комнату».
- **Планирование спринта** — слева бэклог; в центре состав спринта (drag-and-drop) + прогноз влияния; справа загрузка команды, баланс спринта, предупреждения, чат с ассистентом.
- **Ретроспектива** — авто-статистика спринта; интерактивная доска (хорошо / плохо / действия) с голосованием; чат для генерации идей.
- **Груминг** — список задач с индикаторами готовности (макеты, критерии, оценка); быстрое уточнение через ассистента.

### 9.7 База знаний (`/knowledge`)
Поиск по задачам, документам, коду, метрикам, макетам, исследованиям.
- **Граф** — узлы = артефакты (эпик, задача, PR, метрика, Confluence, исследование), связи = зависимости и влияние; двойной клик → карточка.
- **Карточка артефакта** — вкладки: Обзор, Связанные объекты, История изменений; ручное добавление связей.

### 9.8 Центр уведомлений
Выпадающая панель (~400px) из хедера. Уведомления сгруппированы по дате: иконка типа (инцидент, агент, упоминание, результат сервиса), заголовок, время. Фильтры: Все / Непрочитанные / По типу. «Открыть все» → `/notifications`.

### 9.9 Профиль и настройки (`/profile`)
Вкладки:
- **Уведомления** — каналы и тонкая настройка по типам событий.
- **Интеграции** — подключение Jira, Confluence, Git, Figma.
- **Персонализация** — язык, тема, уровень детализации ассистента.
- **Агенты и сервисы** — подписки на агентов, избранные внешние сервисы.
- **Команды и продукты** — управление списком продуктов и доступом.

## 10. Модель данных

Доменные типы (в `data/types/`). На этапе прототипа — это форма фикстур.

- **Product** — id, name, team, метрики.
- **User** — id, name, role, avatar, настройки.
- **Task** — id, title, status, assignee, priority, riskLevel, метрики-связи, прогноз влияния, дедлайн, criteria, complianceChecklist, связанные артефакты.
- **Agent** — id, name, category, описание, isActive, trigger, condition, action, lastFired, eventCount24h, лог.
- **AIService** — id, name, category, описание, usefulnessScore, hasCorpSubscription, стоимость, комментарии, ссылка.
- **Room** — id, type (planning|retro|grooming), статус, дата, участники, данные церемонии.
- **Notification** — id, type (incident|agent|mention|service-result), заголовок, время, isRead.
- **Artifact** — id, type (epic|task|pr|metric|confluence|research), связи (edges) для графа.
- **Dialog / Message** — id, контекст (product/sprint/task), участники, сообщения (роль, контент, вложения).
- **Metric** — id, серия значений, порог, динамика.

## 11. Интеграции (фаза 4)

Внешние источники, которые на проде подключаются через настройки → Интеграции:

- **Jira** — задачи, статусы, спринты (Задачи — надстройка над ним).
- **Confluence** — документация, артефакты в базе знаний.
- **Git** — PR, изменения кода, авто-проверки.
- **Figma** — макеты, дизайн-система, accessibility.
- **Метрики** — воронки, NPS, скорость команды.
- **Поддержка** — тикеты, обращения.

На этапе прототипа все они мокаются. Слой `data/api/` спроектируй так, чтобы реальные адаптеры можно было подставить без изменения UI.

## 12. Ключевые сквозные сценарии

Это «золотые пути» — их прохождение целиком должно работать в прототипе.

### 12.1 Реакция на падение метрики
*Пример: падение конверсии в кредитной заявке мобильного приложения.*
1. **Обнаружение** — агент «Метрический дозорный» фиксирует падение конверсии шага «ввод паспорта» на 10%, находит подозрительный PR #4521, сканирует тикеты и UX-базу, создаёт инцидент в Jira, уведомляет PO.
2. **Анализ и постановка** — PO в ассистенте видит полную картину, создаёт задачу; ассистент генерирует User Story, Impact Map, compliance-чеклист.
3. **Дизайн** — дизайнер получает прототип из дизайн-системы, проверенный на accessibility.
4. **Разработка** — при PR агенты проверяют соответствие макету, архитектуре, безопасности, тестовое покрытие.
5. **Тестирование и приёмка** — ассистент генерирует тест-кейсы, собирает сводку готовности.
6. **Релиз и мониторинг** — агент «Релиз-аналитик» отслеживает метрики после деплоя, публикует отчёт (конверсия +6 п.п., ошибки −70%), сохраняет результат в базе знаний.

### 12.2 Планирование новых фич
1. **Сбор сигналов** — агенты мониторят рынок, поддержку, метрики, технологии → «Банк возможностей».
2. **Брейншторминг** — ассистент-фасилитатор предлагает техники, генерирует идеи, строит mind-map.
3. **Валидация** — через внешние сервисы и внутренние данные оцениваются прогноз влияния, осуществимость, compliance-риски.
4. **Приоритизация** — агент «Приоритизатор» скорит инициативы, формирует драфт роадмапа; PO правит в диалоге.
5. **Декомпозиция** — эпик автоматически разбивается на User Stories с критериями приёмки и привязкой к метрикам.

## 13. План разработки по фазам

**Фаза 0 — каркас.** Vite + TS + antd (тёмная тема через `ConfigProvider` + `darkAlgorithm`) + роутинг, AppShell (хедер, сайдбар), design tokens antd, слой фикстур и `data/api`.

**Фаза 1 — статичные экраны.** Все разделы из навигации в read-only виде на моках: дашборд с виджетами, список диалогов, каталоги агентов и сервисов, три представления задач, список комнат, граф знаний, панель уведомлений, профиль.

**Фаза 2 — интерактив.** Drag-and-drop (Канбан, состав спринта, ретро-доска), переключение представлений, фильтры, контекстные меню, чат ассистента с моковыми ответами, конструктор агентов.

**Фаза 3 — сквозные сценарии.** Сценарии 12.1 и 12.2 проходятся целиком на моках, включая «срабатывание» агента и генерацию артефактов.

**Фаза 4 — реальные данные.** Адаптеры интеграций (Jira, Confluence, Git, Figma, метрики, поддержка), настоящий ассистент и агенты, аутентификация, права доступа.

**Фаза 5 — продакшн.** a11y-аудит, производительность, тесты, наблюдаемость.

## 14. Конвенции для Claude Code

- **Язык интерфейса** — русский. Доменные термины (Канбан, бэклог, груминг, PR, compliance) сохраняем как в спецификации.
- **UI только на Ant Design (antd v5)** — не вводим параллельные UI-библиотеки и не пишем компоненты, которые в antd уже есть.
- **Только тёмная тема** — через `ConfigProvider` + `darkAlgorithm`; стилизация через design tokens, не произвольный CSS поверх antd.
- **TypeScript строгий** — без `any`, доменные типы в `data/types/`.
- **Одна фича — одна папка** в `features/`. Не размазывай логику раздела по проекту.
- **Данные только через `data/api/`** — компоненты не читают фикстуры напрямую, чтобы фаза 4 не ломала UI.
- **Доступность** — antd закрывает базу; проверяем навигацию с клавиатуры, фокус-стили и контраст.
- **Цвета рисков последовательны** — бери из design tokens (`colorError`/`colorWarning`/`colorSuccess`), не хардкодь.
- **Без бизнес-логики в компонентах** — скоринг приоритета, форматирование и расчёты живут в `lib/`.
- **Маленькие компоненты** — если файл разрастается, дроби.
- Перед коммитом — типчек, линт и тесты.

## 15. Деплой и окружения (Vercel)

Хостинг — **Vercel**. Подключаем рано (сразу после каркаса фазы 0), чтобы с самого начала каждая ветка/PR давала preview, а `main` автоматически уходил в production.

**Как это работает:**
- Vercel автоопределяет Vite: framework preset — Vite, build command `npm run build`, output directory `dist`.
- Push в ветку / открытие PR → preview-деплой с уникальным URL (удобно показывать экраны заказчику).
- Merge в `main` → production-деплой.

**SPA-роутинг.** React Router работает на клиенте, поэтому без rewrite прямой заход на `/tasks` вернёт 404. Кладём в корень `vercel.json`:

```json
{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

**Переменные окружения** (понадобятся на фазе 4 для интеграций):
- Клиентские переменные во Vite обязаны иметь префикс `VITE_`, иначе не попадут в бандл; читаем через `import.meta.env.VITE_...`.
- Секреты задаём в настройках проекта Vercel по окружениям (Production / Preview / Development), не в репозитории.

**Деплой из сессии.** К Claude подключён коннектор Vercel — можно деплоить, смотреть список деплоев, читать build-логи и получать ссылки прямо из сессии, не открывая дашборд. Используем его, когда нужно выкатить сборку или разобрать упавший билд.

**Гигиена репозитория:** добавь `.vercel/` (локальная привязка проекта) в `.gitignore`.

**Безопасность:** production-деплой — только по явному подтверждению владельца, не автоматически из сессии.
